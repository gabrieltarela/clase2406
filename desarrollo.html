Identidad y Unicidad

Restricciones del Modelo Relacional

De la Superclave a la Clave Primaria: garantizando la calidad de los datos.

"La base de datos no es solo tablas, son reglas para evitar el error".

¿Qué vamos a lograr hoy?

  • Dominar la lógica de Unicidad (Superclave, Candidata y PK).
  • Entender el Dominio como el primer filtro de calidad.
  • Aprender a elegir la mejor Clave Primaria para un diseño eficiente.

El problema de la identidad

"Si tenemos 100 personas llamadas 'Ana Silva',
¿qué dato garantiza que nunca las confundamos?".

¿Cédula? ¿Email? ¿Número de registro?

1. Restricción de Dominio

Es el "Molde de Calidad": define qué valores son permitidos.

Ejemplos:
  • Edad: Solo números positivos.
  • Fecha: Formato DD/MM/AAAA.

Si el dato no respeta el dominio (ej. una edad negativa), el sistema lo rechaza para mantener la integridad.

2. Superclave

Cualquier conjunto de columnas que identifique a una fila.

Analogía: Es una red de pesca gigante. Atrapa la fila, pero trae datos que "sobran".
Ejemplo: (Cédula + Nombre). Identifica a la persona, pero el Nombre no es necesario si ya tenemos la Cédula.

3. Clave Candidata

Es una Superclave mínima. No tiene datos redundantes.

Minimalidad: Si le quito un solo atributo y ya no puedo identificar a la persona, entonces es una Candidata.
Ejemplos: Cédula, Matrícula Profesional, Email.

4. Clave Primaria (PK)

Es la candidata oficial elegida por el diseñador.

Criterios de elección:
  1. Estabilidad: Que el dato no cambie en el tiempo.
  2. Simplicidad: Que sea fácil de manejar por el sistema.

Notación: Atributo subrayado.

Ejemplo: Staff Médico

Atributos: ID_Interno, Matrícula, Nombre, Especialidad.

  • Superclave: (Matrícula + Nombre).
  • Candidatas: ID_Interno, Matrícula.
  • PK Elegida: ID_Interno (Estable y controlado).

Reto: Tabla Vehículos

Atributos: Matrícula, Nro_Chasis, Marca, Modelo, Color.
